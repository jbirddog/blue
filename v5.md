# Blue v5

## On v4 and a v5

v4 was very close to the goal i think. At this time i think the separations are not quite right.

One key thing though was the realization that Blue at its core it not so much a full featured Forth that has no
stack after compilation, but a high level assember with Forth semantics.

### Likes:

1. The compiler can run any previously compiled code at compile time
1. The compiler works on bytecode

### Dislikes:

1. Implementing Blue on BlueVM ended up having the frontend keep track of the stack state

## New Direction

1. v5 is more of an integrated environment which works on a binary block file structure
1. Like colorForth this file is a mix of object code and bytecode
1. The core system is a very small loop that interprets the binary file
1. Part of these blocks is an editor which can display/edit the block file
1. Code buffer can be saved as an external file when compilation is done if desired
1. By default you are in a repl
1. Some block will run by default after the system boots
1. 

### Editor

1. Have always liked the ColorForth editor concept - it does all the parsing, etc
1. Have thought about doing an editor several times but keep punting on it
1. The ColorForth editor works on binary files
   1. This is hard on version control
   1. Do believe this is the correct input to the compiler
   1. Allows using traditional blocks but is storing compiled tokens vs raw text input
   1. Something like the Redis protocol
   1. Just have a separate tool that encodes/decodes this textual representation
   1. This tool can help bootstrap some binary files for the editor to display
