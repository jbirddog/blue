# Blue v5

## On v4 and a v5

v4 was very close to the goal i think. At this time i think the separations are not quite right.

One key thing though was the realization that Blue at its core it not so much a full featured Forth that has no
stack after compilation, but a high level assember with Forth semantics that can execute any previously assembled
code.

Would also like to take v5 as an opportunity to not be x86_64 specific.

## C Prototype

1. Have two values on the data stack
1. Compile machine code to push all registers that will be used
1. Compile machine code to "flow in" those values and `add`
1. Compile machine code to "flow out" the result and push it on the data stack
1. Compile machine code to pop all pushed registers
1. Call this machine code and return back to C
1. Think this will be all that is needed for the POC

Prototype worked, small learnings:

1. Don't have to push/pop all registers that will be used, just respect C calling convention
1. Due to C calling convention was able to pass in `tos` and have it in rdi, so can `stosq` to push outputs

### Likes:

1. The compiler can run any previously compiled code at compile time
1. The compiler works on bytecode
1. Frontend that generates this bytecode
1. Frontend contains the dictionary
1. Compiler never does lookups, just works on offsets
1. Running dup/drop/b, etc at compile time

### Dislikes:

1. Implementing Blue on BlueVM ended up having the frontend keep track of the stack state

## New Direction

1. Instead of the bytecode targeting the VM, have the main goal be simplifying the compiler
1. Frontend is basically tokenizing and mapping to bytecode
1. Compiler is basically interpreting bytecode to build the code buffer
