# v2 Ideas/Changes

## Keep

* stack does not exist after compile time
* developer should have full access to registers/memory as if writing asm
* still x8664/linux centric but keep in mind ways to change that
* still relying on an external assembler for now

## Change/Improve

* reduce the need to declare registers
* opcodes need to be defined as "normal" words instead of being built into the compiler
* no opcodes/syscalls are defined by default
* words have access to the compilation buffer, first pass will continue to write asm strings
  * fasm/nasm
  * can migrate by writing `db`'s directly to the file  
* remove the optimizer since some language features rely on it
* remove the concept of indirect registers
* add ability to push/pop from the cpu stack (return stack)
* some form of overloading/multiple dispatch needs to be supported per word
* some form of type checking the stack vs called word
* simplify the parser
* quickly level up for leverage

## Overloading/Mutliple Dispatch

consider a subset of `add`:

| Opcode | Instruction |
|----|----|
| 04 ib | ADD AL, imm8 |
| 05 id | ADD EAX, imm32 |
| 00 /r | r/m8, r8 |
| REX.W + 01 /r | ADD r/m64, r64 |

There are 22 other variations, most opcodes have a similar number of variants. Adding if/else code on the stack values 
would be cumbersome as would adding N words with variant names (add/1, add/2, add_r/m8,r8, etc). Adding some form of 
overloading/multiple dispatch would simplify work for the word author and caller.

This likely goes hand in hand with type checking the stack when a word is called. In the current version of `Blue` a 
word would be defined as:

`: word ( arg1:rax arg2 -- res:eax ) ... ;`

Where the value after a `:` indicates the register, stack arguments without a `:` are inferred based on the stack affect 
of its called words. This approach likely is not far off from what v2 needs except the register names are assumed and 
static. Instead types with some mask should be defined at design time of the application.

Since words can be overriden, a new defining word should exist for overloading/multiple dispatch:

```
*: add ( dest:al src:imm8 -- al ) ... ;
*: add ( dest:eax src:imm32 -- eax ) ... ;
*: add ( dest:r/m8 src:r8 -- res:r/m8 ) ... ;
*: add ( dest:r/m64 src:r64 -- res:r/m64 ) ... ;
```

_Note: syntax is WIP._

The result of add lands in the `dest` argument. Repeating the type is not horrific but could be `dest` if desired.

* if a `:` is used to redefine a word that was previously defined with `*:` all overloades are forgotten
* if `*:` is used to declare a word for the first time it works the same as a single `:`
* likewise a `*:` appends an overload, so can be followed by a previous `:` definition

## Types (todo: better term)

* calling these `types` is a bit loose in the typical sense. A lot of this has to do with how/where the data is stored
* if types are encoded in a way that size is specified that would be beneficial
* r64 is qword size, register type, no value
* r32 is dword size, register type, no value
* eax is dword size, register type, "eax" value
* 0 is dword? size, imm32 type, 0 value

The current version of `Blue` has no way to define types at design time. `rax`, `edi`, `r8` etc are hardcoded in the 
compiler. There is no way to specify that something is in memory, only in registers. In order to support something like 
the `add` example above, types need to be declared at design time and used:

```
1 type AL
2 type imm8
3 type EAX
4 type imm32
5 type r/m8
6 type r8
7 type r/m64
8 type r64
```
_Note: syntax is WIP and this is over simplified._

Now that these types are defined, the compiler can find a word with some name and some signature. Likely expressing 
types as uint64 will be simple enough and cover normal cases. This will not provide some super amazing type system 
which is ok.

## Moves

The current version of `Blue` inserts `mov` instructions befor calling a word if the word's stack effect specify 
different registers than those currently in play. This will no longer work as expected if the stack effect types are 
defined at application design time. Likely this will require a hook into the compiler that is set to specify how 
different moves should be compiled, eg: 

```
*: mov ( a:r64 b:imm32 -- ) ... ;

' mov hook move
```

## Simplify Parsing Stack Effect Comments

If types can be defined at application design time then the current stack effect comments of `Blue`:

```
: abc ( a b -- c ) ... ;
: abc ( a b:imm8 -- imm32 ) ... ;
```

Will pose a parsing problem. In the current parser this is ok because the available types are hardcoded and checked. 
However now a type `c` could have been introduced, or introducing a new type high up could break existing code.

_Note: I also never really liked the custom parsing of the stack effect names/types._

The requirements are:

* a name, type or both can be specified for each stack slot
* if no type is provided it is inferred if possible, else compile error
* if no name is given, ok
* should not eat up a precious character for later syntax
* ideally the most likely scenario is not explicitly typed and inferred

Some alternate syntax options:

```
: abc ( a b ` imm8 -- ` imm32 ) ... ;

: abc ( a b -- c ) ` _ imm8 -- imm32 ` ... ;
: abc ( a b -- c ) ` ? imm8 -- imm32 ` ... ;

: abc t( a ? b imm8 -- c imm32 ) ... ;
: abc (( a (_) b imm8 -- c imm32 )) ... ;
: abc (( a # b imm8 -- c imm32 )) ... ;
: abc #( a # b imm8 -- c imm32 )# ... ;
```

_Note: leaning toward one of the last two. `(( ))` may be faster to type than `#( )#`. `#` is a "type hole" that needs to be filled.

## Showing Values

At least while we are emitting asm textually, the ability to "show" a stack value will be helpful for the words that write to the 
compilation buffer. For example an implementation of `add`:

```
*: add (( src imm32 -- res eax )) ... ;
*: add (( src imm32 dest r32 -- res r32 )) ... ;
```

Would need to be able to "show" the immediate value and the name of the register. Doing so would consume the stack items:

```
*: add (( src imm32 -- res eax )) show f" add eax, %" ... ;
*: add (( src imm32 dest r32 -- res r32 )) show swap show swap f" add %, %" ... ;
```

"show" is a compile time word that consumes a stack item and places an immediate string representation. `f"` is a compile 
time word that consumes a number of stack items corresponding to the number of `%` and builds a string.

Once the decision is made to drop the dependency on an external assembler, "show" can either be dropped or replaced with an 
appropriate version that emits the required binary code.

## Quickly Leverage Higher Constructs

In the above show example there is a `show swap show swap`. Factor did an amazing job at providing higher level words to help in 
situations such as this. I may not fully remember how they work but it is close to `[ show ] bi`. Will need to review Factor again 
and take ideas, truly a beautiful language. `[ x ]` is an anonymous function and if it includes just one word a simple compiler 
optimization and be implemented.
